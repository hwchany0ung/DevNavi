## 7. 교훈 및 회고

### 7.1 잘한 점 (Keep)

1. **명확한 설계 문서** (Design Option B)
   - 5계층 방어 구조를 Plan 단계에서 명시
   - M3 burst limit 추가 시 기존 체계에 자연스럽게 통합

2. **사전 보안 검토**
   - `/teaser` 엔드포인트 rate limit 누락을 조기에 발견
   - 배포 전 수정으로 운영 문제 예방

3. **포괄적 테스트 계획**
   - 88.8% 커버리지로 동적 필드(M4)도 발견
   - 기존 테스트 재검토로 새로운 문제 해결

### 7.2 개선할 점 (Problem)

1. **초기 설계 완전성**
   - M2: user_id 타입을 Optional으로 설정했으나 비즈니스 로직상 필수
   - 요구사항 명확화 단계에서 "필수 vs 선택" 재확인 필요

2. **엔드포인트 리뷰 누락**
   - M3: `/teaser` rate limit이 다른 엔드포인트에는 있으나 이 곳에만 빠짐
   - Design 단계에서 모든 엔드포인트 일괄 검토 필요

3. **테스트 동기화 시점**
   - M4: 구현 후 테스트 실행 시 컬럼명 불일치 발견
   - 동적 스키마는 구현과 동시에 테스트 갱신 필요

### 7.3 다음에 시도할 것 (Try)

1. **Pre-Implementation Checklist**
   - Design 문서 → Implementation Checklist로 자동 변환
   - 모든 엔드포인트, 필드, 함수를 체크리스트화
   - 구현 완료 후 체크리스트로 검증

2. **Test-First 확대**
   - 동적 필드는 테스트 먼저 작성
   - 스키마 변경 시 테스트 자동 갱신 (스크립트화)

3. **설계 리뷰 Gate**
   - Plan/Design 단계에서 security-architect 필수 검수
   - "모든 입력점에 rate limit이 있는가?" 같은 체크리스트
